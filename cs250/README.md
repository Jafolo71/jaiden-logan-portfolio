This folder contains my Sprint Review and Retrospective from the CS 250 final project (SNHU Travel).

## User needs / user stories
I interpret user needs by turning them into clear, testable requirements. 
User stories help because they simplify what the user wants and make it easier to define what “done” looks like so I’m not guessing while building.

## How I develop programs / Agile processes I’ll use
I prefer working in small chunks and checking progress often instead of doing everything at once. 
In the future I want to keep using a backlog/board, clear “definition of done,” and short reviews/retrospectives to reduce rework when requirements change.

## Being a good team member
Being a good team member means communicating clearly, asking questions early when something is unclear, and keeping work visible so the team stays aligned.
